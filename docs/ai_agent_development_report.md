# LLM・AIエージェント活用による校内QRクイズラリー開発レポート

## 1. 概要

2026年5月19日の台湾交流会に向けて、池田市立石橋小学校と嘉義縣文昌國民小學の児童が共同で楽しめる「校内QRクイズラリー」Webアプリを短期間で構築した。

児童は班ごとにiPadを1台持ち、校内に掲示されたQRコードをiPad標準カメラで読み取り、Safariで開いた問題画面から4択クイズに答える。問題は、日本語で台湾や文昌國民小學について扱うJ問題と、繁体字中国語で日本や石橋小学校について扱うC問題に分かれる。日本側・台湾側の児童が互いに聞き合い、言語や文化を横断して協力することを活動の中心に置いた。

アプリ本体、問題データ、QR生成、印刷物、説明資料、メール文面、配布資料、ロゴ、GitHub Pagesデプロイまでを、ChatGPTとCodexを役割分担して進めた。

## 2. 目的

- 交流会当日に、日本側・台湾側の児童が自然に会話するきっかけを作る。
- 言語差を障壁ではなく、協力の理由に変える。
- 教師個人のレベルで、短期間にWebアプリ、QR、印刷物、説明資料を作成できるか検証する。
- LLMとAIエージェントを、単なる文章作成ではなく、企画・実装・運用準備まで一連のプロセスで活用する。

## 5. 技術構成

- フロントエンド: React + TypeScript + Vite
- 状態保存: LocalStorage
- デプロイ: GitHub Pages
- CI/CD: GitHub Actions
- QR生成: Node.js + `qrcode`
- PDF生成: Node.js + `pdfkit`
- PowerPoint生成: `pptxgenjs`
- 問題入力: Excelテンプレート + Python取り込みスクリプト

外部サーバーやデータベースは使わず、静的Webアプリとして構築した。学校iPadの外部アクセス制限を考慮し、早期に仮デプロイURLを作り、アクセス確認と許可申請を優先した。

## 6. 開発の流れ

### 6.1 MVP

最初に、QR読み取り機能をアプリ内に実装しない方針を決めた。iPad標準カメラでQRを読み取り、SafariでURLを開く運用とした。

MVPでは次を実装した。

- `?q=J01` から問題IDを読み取る
- `questions.json` から問題を表示する
- 班名、点数、回答履歴をLocalStorageに保存する
- 同じ問題に一度しか回答できない
- 回答前に確認ダイアログを出す
- 回答後に正解・不正解、獲得点、解説を表示する
- Result画面を表示する
- 先生用リセットを二重確認にする

### 6.2 LocalStorageの堅牢化

レビューにより、iPad Safariのプライベートブラウズやストレージ制限でLocalStorageが例外を投げる可能性が指摘された。これを受けて、`getItem`、`setItem`、`removeItem` をすべて `try/catch` で扱うようにした。

また、Safariで複数タブになる可能性を考慮し、回答送信時に保存済みの最新状態を再読込してからマージする処理を追加した。`storage` イベントも監視し、別タブの状態更新を反映するようにした。

### 6.3 問題差し替えへの対応

本番問題は後からExcelで作成する前提だったため、問題データとアプリ本体を分離した。さらに、`questions.json` の内容から簡易signatureを作り、問題セットが変わった場合は古い回答履歴を自動破棄するようにした。

これにより、ダミー問題から本番問題へ差し替えても、古い回答済み状態が新しい問題をブロックしない構造になった。

### 6.4 翻訳の鍵と宝箱QR

台湾交流会の性質に合わせ、単なるヒントではなく、繁体字中国語・台湾華語への翻訳を有限回だけ見られる「翻訳の鍵」を実装した。

- 初期状態で翻訳の鍵を3本付与
- 翻訳の鍵を使うと該当問題の翻訳を表示
- 点数は減らさない
- 使用履歴をLocalStorageに保存
- 宝箱QRを見つけると翻訳の鍵を1本増やす

この機能は、児童がすぐ翻訳に頼るのではなく、まずチームメイトに聞く設計にした。

### 6.5 QR URLの秘匿化

当初の `?q=J01` では、勘のよい児童がURLから問題番号を推測できる可能性があった。そこで、本番QRには `?q=r15hn85q` のような公開用コードを出力するようにした。

一方で、教師の確認用には `?q=J01` の直接IDも残した。これはデバッグ用エイリアスであり、QRカードやQR一覧には出力しない。

### 6.6 印刷物生成

Webアプリだけでは当日運用が不安定になるため、QRカード、紙掲示用問題、紙バックアップ用回答用紙をPDF生成できるようにした。

- QRカードはA4に2枚、A5程度の大きさ
- 問題QRカードには問題文を載せず、ID・点数・QR・URLを表示
- 宝箱QRカードは特別なデザイン
- 紙掲示用PDFには問題文と選択肢を全文表示
- 紙バックアップ回答用紙にはチーム名、問題ID、答え、得点、先生チェック欄を配置

### 6.7 本番問題取り込み

Excelテンプレートを作成し、そこに本番問題を入力した。追加された `translationExplanation` 列にも対応し、正解・不正解画面で解説翻訳を表示できるようにした。

問題取り込み後、いくつかの問題で正答ズレや表記ズレが見つかり、Excel修正版を再取り込みした。これは、本番前の検証で最も重要な工程だった。

### 6.8 デザインと資料

日本と台湾の交流会らしい明るいUIに整え、さらに「石小×文昌」のロゴを作成してアプリや資料に反映した。

また、児童向け説明PowerPoint、日本語・英語説明原稿、台湾児童向け繁体字ルール配布資料、相手校へのメール文面と添付資料も作成した。

## 7. Gitの活用

今回の開発では、変更ごとにcommitを残した。主な流れは次の通り。

- `Initial Vite React scaffold`
- `Harden local storage and question set handling`
- `Add translation keys and treasure QR rewards`
- `Add print materials and exchange themed design`
- `Prepare GitHub Pages deployment`
- `Add bilingual UI and timed answer flow`
- `Use opaque QR URL codes`
- `Add Wenchang quiz request materials`
- `Add question replacement Excel template`
- `Import production quiz data`

Gitを使ったことで、直前のバグ修正も小さな差分として管理できた。特に本番直前は、どの修正が入ったか、GitHub Pagesに反映されたかを確認しながら進められた。

## 8. 当日運用

当日は、児童が班ごとにiPadを持ち、校内に貼られたQRコードを探してクイズに答えた。終了後はResult画面をスクリーンショットし、ロイロノートで提出する運用とした。

日本側・台湾側の児童が互いに問題の意味や文化的背景を聞き合う場面があり、活動としては成功した。

## 9. 発生した課題

### 9.1 iOSコントロールセンターQRリーダー問題

iPad標準カメラでは正常にLocalStorageへ点数が保存されたが、iOSのコントロールセンターにあるQRリーダーから開いた場合、点数が最後に0になった端末があった。

原因として、Safari本体とは異なる一時的なWebViewで開かれ、LocalStorageが共有・保持されなかった可能性が高い。

今後は次を徹底する。

- QRカードに「iPad標準カメラで読む」と明記する
- 「コントロールセンターQRリーダーは使わない」と明記する
- 説明スライド・児童説明原稿にも入れる
- 開始前に、標準カメラで読み取る実演をする

### 9.2 端末依存の保存リスク

LocalStorageは軽量で静的アプリに向く一方、ブラウザやWebViewが変わると保存が分かれる。ランキング自動集計をしない今回の設計では十分だったが、今後は必要に応じてGoogleフォーム、GAS、Firebase、Supabaseなどを検討する余地がある。

## 10. LLM・AIエージェント活用の知見

### 10.1 ChatGPTは企画・言語・表現に強い

ChatGPTは、活動意図の整理、メール文面、説明資料、ロゴのイメージ作り、翻訳の文面作成に有効だった。相手校に送る文章や児童向け説明など、文脈とトーンを整える場面で特に役立った。

### 10.2 Codexは実装・検証・Git運用に強い

Codexは、リポジトリ内のファイルを読み、既存構造に合わせて実装し、lint/buildを回し、commit/pushまで行える点が有効だった。単なるコード生成ではなく、プロジェクト全体の編集と検証を継続的に任せられた。

### 10.3 教師がやるべき判断は残る

AIは実装や文案を高速化できるが、活動目的、校内事情、児童の実態、当日の動線、安全判断、本番で使えるかどうかの最終判断は教師が担う必要がある。

## 11. 今後に向けた標準フォーマット

同様のプロジェクトを行う場合、次の流れが有効である。

1. 活動目的を1文で定義する。
2. 参加者、端末、ネットワーク、時間、場所の制約を列挙する。
3. MVPを決める。
4. データとアプリ本体を分離する。
5. 早めに仮URLを作り、学校端末で確認する。
6. 紙バックアップを作る。
7. 本番データ差し替え手順を作る。
8. 印刷物を自動生成する。
9. 児童向け説明資料を作る。
10. 前日までにQR実機確認をする。
11. 当日のトラブルを記録し、次回版へ反映する。

## 12. ChatGPT共有リンクを読めない場合の補助プロンプト

CodexがChatGPT共有リンクを直接読めない環境では、ChatGPT側で次のプロンプトを使って、会話の要約を生成するとよい。

```text
以下の共有会話を、後から別のAIエージェントや開発者が読んで理解できるように整理してください。

目的:
- 台湾交流会QRクイズラリーの発案からCodexへ実装依頼するまでの流れを記録する
- 動機、教育的ねらい、制約、最初に検討した案、採用した案、見送った案を整理する
- 後続の開発レポートに貼れるように、時系列と論点別の両方でまとめる

出力してほしい内容:
1. 背景と動機
2. 活動アイデアが固まるまでの流れ
3. Webアプリ化を選んだ理由
4. QRクイズラリー案の要点
5. 台湾交流会として大事にした教育的価値
6. 実装前に決めた制約
7. Codexに渡す時点での要件
8. 今後同様の企画に活かせる知見

文体:
- 日本語
- 中程度以上の専門性がある教員・情報担当者向け
- 箇条書き中心で、後からレポートに統合しやすい形
```

## 13. 結論

今回の実践は、教師個人でも、LLMとAIエージェントを使えば、交流活動に合わせたWebアプリと運用資料を短期間で構築できることを示した。

重要なのは、AIに丸投げすることではなく、教師が活動目的と制約を明確にし、AIに具体的な作業を委ね、検証と運用判断を人間が担うことである。
