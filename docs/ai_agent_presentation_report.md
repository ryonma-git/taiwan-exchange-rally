# 学校現場におけるLLM・AIエージェント活用事例

## 事例名

台湾交流会「校内QRクイズラリー」Webアプリの短期開発と当日運用

## 対象読者

- 学校の情報担当者
- ICT活用推進担当
- 校務や授業でAI活用を検討している教職員
- 教育委員会・研究会・情報部会の参加者

## 1. 何を作ったか

台湾の小学校との交流会で使う、校内QRクイズラリー用のWebアプリを作成した。

児童は班ごとにiPadを持ち、校内に貼られたQRコードを探す。QRをiPad標準カメラで読み取り、Safariで開くと4択問題が表示される。正解すると点数が加算され、最後にResult画面をスクリーンショットしてロイロノートで提出する。

問題は2種類。

- J問題: 日本語で出題。内容は台湾、嘉義、文昌國民小學。
- C問題: 繁体字中国語で出題。内容は日本、大阪、石橋小学校。

日本側と台湾側の児童が互いに聞き合わないと解きにくい構造にした。

## 2. なぜ作ったか

交流会では、言語の違いがあるため、児童同士の会話が生まれにくい可能性がある。そこで、言語差を「困りごと」ではなく「協力する理由」に変える活動として、QRクイズラリーを設計した。

Webアプリ化した理由は次の通り。

- iPadだけで実施できる
- 紙だけよりゲーム性がある
- 得点が自動で累積される
- QRを校内に貼ることで移動と探索が生まれる
- 問題データを後から差し替えやすい
- 静的Webアプリなら無料または低コストで公開できる

## 3. AIをどう使ったか

### ChatGPT

主に企画・文章・表現に使った。

- 活動案の壁打ち
- 台湾交流会としての教育的価値の整理
- 相手校へのメール文面
- 児童向け説明原稿
- ロゴ生成のためのプロンプト
- 繁体字資料の文案

### Codex

主に実装・検証・Git操作に使った。

- React + TypeScript + Vite のアプリ実装
- LocalStorage保存
- QR URL処理
- 翻訳の鍵・宝箱QR
- タイマー
- GitHub Pagesデプロイ
- QRカードPDF生成
- 紙掲示用PDF生成
- Excelテンプレートと問題取り込み
- lint/build確認
- commit/push

## 4. 開発フロー

1. 活動目的を整理
2. MVPを決定
3. React静的Webアプリを作成
4. 問題データをJSONに分離
5. 仮デプロイURLを作成
6. 学校iPadでアクセス確認
7. QR URLとPDFを自動生成
8. 本番問題をExcelで作成
9. ExcelからJSONへ取り込み
10. QRカードと紙問題を印刷
11. 説明資料と配布資料を作成
12. 当日運用
13. 課題を次回版へ反映

## 5. 技術的なポイント

### 静的Webアプリにした

サーバーやデータベースを持たず、GitHub Pagesで公開した。これにより、構築と運用が軽くなった。

### LocalStorageを使った

班名、点数、回答履歴、翻訳の鍵の使用状況を端末内に保存した。1班1台で運用する前提なら十分実用的だった。

### 問題データを分離した

問題文、選択肢、正解、解説、翻訳は `questions.json` に持たせた。後から本番問題へ差し替えられるようにした。

### QR URLを推測しにくくした

内部IDは `J01`、`C01` だが、印刷用QRには `r15hn85q` のような公開用コードを使った。児童がURLから問題番号を推測しにくくした。

### 紙バックアップも作った

Webがブロックされた場合に備え、回答用紙PDFを作成した。学校現場では、デジタルだけでなく紙の逃げ道が重要である。

## 6. 当日の結果

活動は成功した。日本側・台湾側の児童がクイズを楽しみ、互いに聞き合いながら進めることができた。QRコードも基本的には正常に動作した。

ロゴは手紙の裏面にも活用され、交流の記念品としても役立った。

## 7. 失敗・反省

iOSのコントロールセンターにあるQRリーダーから開いた場合、点数が保存されず、最後に0点になる端末があった。

原因は、Safari本体とは別の一時的なWebViewで開かれ、LocalStorageが共有されなかった可能性が高い。

次回は次を明記する。

- QRはiPad標準カメラで読む
- Safariで開く
- コントロールセンターQRリーダーは使わない
- 開始前に実演する

## 8. AI活用の価値

今回の重要なポイントは、AIが「文章を書く道具」だけではなかったことである。

Codexは、リポジトリを読み、既存の構造を理解し、実装し、テストし、Gitにcommitし、GitHub Pagesへ反映するところまで担当できた。これにより、教師は活動の目的、児童の実態、学校の制約、当日の判断に集中できた。

## 9. AI活用の注意点

- AIの提案は必ず現場の制約で検証する。
- 本番前に実機で確認する。
- Gitで変更履歴を残す。
- 紙バックアップを用意する。
- 児童の個人情報を扱う場合は、保存先と共有範囲を慎重に検討する。
- 多言語資料は、可能なら母語話者・相手校に確認してもらう。

## 10. 他校で再利用する場合のテンプレート

### 企画テンプレート

- 活動名
- 対象学年
- 参加人数
- 使用端末
- ネットワーク制約
- 活動時間
- 探索範囲
- 成績の集め方
- トラブル時の紙運用
- 児童に期待する対話

### 開発テンプレート

- MVPを決める
- データとアプリを分離する
- 早めに仮デプロイする
- 実機確認する
- QRと印刷物を自動生成する
- Gitで差分を残す
- 本番前チェックリストを作る

## 11. 発表で強調したいこと

教師がAIを使う価値は、単に作業時間を短縮することではない。

本来なら「時間がなくてできない」と諦めていた活動を、現場の意図に合わせて具体化できる点にある。AIエージェントを使えば、アイデア、実装、資料、印刷物、運用改善までを一つの流れとして扱える。

ただし、活動の目的を決めるのは教師であり、最終的な安全確認と運用判断も教師が担う。この分担がうまくいくと、学校現場でも個別最適な小さなシステムを作れる。

## 12. 一言でまとめると

LLMとAIエージェントは、学校現場で「自分たちの活動にぴったり合う小さなアプリ」を作るための実用的な道具になり得る。
